Please note that:

1. ```getLockedCount()``` function is refactored to use the new mapping '_lockedDelegations[holder]' to keep track of locked delegations. Delegations are added to '_lockedDelegations' upon creation and removed upon unlock. Thus, the function does not have to loop through all delegations making it gas effective.

2. ```endingDelegatedToUnlocked()``` function has been refactored to implement a pull over push paradigm. This mitigates the susceptibility to re-entrancy attacks by ensuring the state of the smart contract is updated before any funds are transferred. Only after successful update of state variables, the payout is enabled but it's the user's responsibility to handle the withdrawal of funds. Instead of immediately transferring funds, the potential for withdrawal is logged in the '_totalDelegated' and '_purchased' state variables. The actual payout can be handled in a separate withdrawal function, not depicted in the provided JSON, since it is not directly part of the functions being addressed.

Please note, while this protects against the described vulnerabilities, a comprehensive audit might reveal further issues not addressed in this “spot-fix”. It’s also generally suggested to adopt the latest stable Solidity version where possible, to benefit from more recent security measures, and optimizations.
