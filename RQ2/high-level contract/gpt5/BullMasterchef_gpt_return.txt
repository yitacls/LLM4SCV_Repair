    "description": "The _massUpdatePools function iterates through all pools and calls _updatePool for each one. As the number of pools (poolInfo.length) increases, the gas cost of this loop grows linearly. If the total gas required to iterate all pools exceeds the block gas limit, any transaction that calls _massUpdatePools will run out of gas and revert. In this contract _massUpdatePools is invoked by owner-controlled functions add, set and updateEmissionRate. Consequences:\n\n- Denial of Service: When pool count is large enough, the owner will no longer be able to call add, set or updateEmissionRate because these functions call _massUpdatePools and will revert due to gas limits. This prevents adding or updating pools or changing emission rates.\n\n- Potential functional impact: While per-pool interactions that do not call _massUpdatePools (e.g. user deposit/withdraw which call _updatePool for a single pid) may still work, administrative operations become unusable and reward accounting that relies on mass updates may be hindered. If the project relied on the ability to change allocations or emissions in response to issues, those mitigations become impossible.\n\nAttack / misuse scenario: An attacker (or simply natural growth of pools) causes pool count to grow such that _massUpdatePools exceeds block gas limit. This makes owner-only functions that call it unusable, effectively locking contract configuration and possibly harming the protocol.\n\nRecommendation (not exhaustive): Avoid unbounded loops that depend on a mutable array length in functions callable on-chain. Instead use patterns that update pools lazily (per-pool updates), allow partial batch updates with explicit indices, or restrict the number of pools and avoid calling a full mass-update inside functions that need to remain callable."
