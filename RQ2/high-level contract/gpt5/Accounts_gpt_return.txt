  "description": "The contract iterates over the full token registry length (tokenNum / coinLength) in several public/external functions (getBorrowETH, getDepositETH, isAccountLiquidatable, claim, etc.). These loops perform external calls and per-token computations for each index. If the token registry grows large (or if an attacker or privileged party can increase the registry size), these loops can exceed the block gas limit and cause the functions to run out of gas and revert. This results in a Denial-of-Service (DoS) for functionality that relies on these functions (e.g., liquidation checks, claiming FIN, computing balances), potentially preventing normal protocol operations.\n\nImpacts:\n- Critical functions may become uncallable when tokenNum is large, preventing liquidations, claims, borrowing/repaying flows, and balance queries.\n- An adversary or misconfiguration that increases registry size can intentionally or accidentally induce DoS.\n\nNotes on exploitability:\n- Even if adding tokens requires privileged access, the vulnerability remains dangerous (administrator error or upgrade could inadvertently enable DoS).\n- The loops also call external contract functions (priceFromIndex, addressFromIndex, bank checkpoints), amplifying gas cost.\n\nRecommended mitigations:\n- Avoid scanning the full registry in on-chain code. Maintain per-account active-token lists (only tokens the user interacts with) and iterate those smaller lists.\n- Limit the maximum token registry length enforced on-chain or limit the number of tokens processed per transaction (use pagination / batched processing).\n- For operations needing full-scope checks (e.g., liquidation eligibility), perform off-chain scanning and submit necessary proofs/targets on-chain, or implement incremental / permissioned liquidation flows that do not require looping through the entire registry in a single transaction."
